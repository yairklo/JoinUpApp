## 1. TECH STACK & ARCHITECTURAL CONVENTIONS

### 1.1 Monorepo layout & runtime

| Layer | Path | Runtime | Module system |
|-------|------|---------|---------------|
| **Express API backend** | `server/` | Node.js | CommonJS (`require`) |
| **Next.js PWA frontend** | `next_app/` | Next.js 15 App Router, React 19, TypeScript | ESM via Next |
| **Prisma schema** | `server/prisma/schema.prisma` | Prisma 6 ORM → PostgreSQL (Neon) | — |
| **Real-time** | `server/index.js` (Socket.io 4) | Live badges, roster sync, chat delivery | — |
| **Authentication** | Clerk (managed) | `@clerk/express` + `@clerk/backend` on server; `@clerk/nextjs` on web | — |

### 1.2 Backend (Express) conventions

- **Entry:** `server/index.js` — CORS, Socket.io, route mounting, chat/game broadcast handlers.
- **Auth middleware:** `server/utils/auth.js` — `authenticateToken`, `attachOptionalUser`, exported `clerkClient`.
- **Domain services:** `server/services/gameService.js`, `server/services/notificationService.js`.
- **Routes:** `server/routes/*` (users, games, series, chats, etc.).
- **Do not extend legacy dead code:** `server/routes/auth.js` and `server/utils/dataManager.js` are legacy JSON-file stores, disconnected from Clerk/Prisma. Never reference or extend them.

### 1.3 Frontend (Next.js App Router) conventions

- **Framework:** Next.js 15 App Router PWA wrapped by `next-pwa`; React 19; TypeScript.
- **Styling triple-stack:** Tailwind CSS v4 + MUI (Emotion) + Bootstrap 5 CSS (legacy navbar). New UI should prefer Tailwind + Radix/CVA; avoid mixing spacing on one element across MUI `sx`, Bootstrap, and Tailwind.
- **No Axios / React Query / SWR.** Client HTTP goes through `apiClient<T>()` in `next_app/src/services/api/client.ts`, with per-domain wrappers under `next_app/src/services/api/{games,users,chats,series,fields,notifications,ratings,search}.ts`.
- **Env URLs:** Never hardcode API or socket origins. Use `process.env.NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_SOCKET_URL` (resolved in `client.ts` and `SocketContext.tsx`).

#### Mandatory provider nesting (`next_app/src/app/layout.tsx`)

Exactly one app-wide tree — never duplicate providers or mount a second `FloatingChatWindow` on a page:

```
ClerkProvider → SocketProvider → ThemeRegistry (MUI) → ChatProvider → NotificationCountersProvider
  → { AppNavbar, <page>, FloatingChatWindow, NotificationAsker }
```

Components using `useChat()`, `useSocket()`, or `useNotificationCounters()` assume this tree.

#### Data-fetching rules

1. **Server Components** fetch with `fetch(..., { cache: "no-store" })` for first paint (e.g. `games/[id]/page.tsx` passes `initialGame` props).
2. **Client calls** use `apiClient()`; all game payloads pass through `mapGameTimezones()` which backfills `date`/`time` from ISO `start` timestamps.
3. **Date formatting:** Use `formatJerusalemDate` / `formatJerusalemTime` from `next_app/src/utils/timezone.ts` with explicit `timeZone: 'Asia/Jerusalem'`. Never hand-roll a second formatter.
4. **Hydration safety:** Never call `new Date()`, `.toLocaleDateString()`, or `.toLocaleTimeString()` in a Client Component render body without explicit timezone or a post-mount gate (`mounted` flag / `useEffect`). Known risk files: `JoinGameButton.tsx`, `Chat.tsx`, `SeriesManager.tsx`, `TeamBuilderWrapper.tsx`, `GamesDateNav.tsx`.
5. **Leaflet maps:** Always `dynamic(..., { ssr: false })` — Leaflet touches `window` at import time (`SearchMapComponent`, `MapComponent`, `GameLocationMap`).

#### Dangerous UI / config changes (require deliberate review)

- Editing `next_app/tailwind.config.ts` or `next_app/src/app/globals.css` CSS variables (`--background`, `--primary`, etc.) — affects Tailwind and MUI globally.
- Bypassing or duplicating the root layout provider tree.
- Editing `next_app/next.config.ts` PWA flags without checking generated `public/sw.js`.
- Note: `eslint.ignoreDuringBuilds: true` — **`next build` succeeding does not prove lint-clean.**

## 2. DATABASE RELATIONSHIPS & INVARIANTS

### 2.1 Neon connection topology (critical)

`server/prisma/schema.prisma` defines:

- `url = env("DATABASE_URL")` — **pooled** traffic via PgBouncer (`-pooler` hostname suffix, e.g. `ep-xxxx-pooler.aws.neon.tech`).
- `directUrl = env("DIRECT_DATABASE_URL")` — **direct** connection for migrations (no `-pooler` suffix).

**Rules:**

1. Hostname distinguishes pool vs direct — **not** port (both use 5432).
2. `prisma migrate dev` / `migrate deploy` require session-level features PgBouncer rejects; Prisma CLI routes migrations via `directUrl`.
3. **Never** point `DIRECT_DATABASE_URL` at a `-pooler` hostname in any environment.

### 2.2 PrismaClient lifecycle (known architectural debt)

- Many modules declare `const prisma = new PrismaClient()` at module scope (~20 instances today).
- **When editing existing files:** reuse the existing module-level instance.
- **Never** instantiate `new PrismaClient()` inside an Express request handler, loop, or worker tick — this exhausts Neon connection limits under load.

### 2.3 Transaction conventions (high-concurrency writes)

Roster joins, waitlist promotion, and game creation **must be atomic**.

**Game + group chat pairing:** When creating a game, create its matching `ChatRoom` in the same interactive transaction. The group chat **`id` must equal the `game.id` exactly**:

```javascript
const created = await prisma.$transaction(async (tx) => {
  const game = await tx.game.create({ data: { /* ... */ } });
  await tx.chatRoom.create({ data: { id: game.id, type: 'GROUP' } });
  return game;
});
```

**Rule:** Every call inside the callback uses `tx.*` only. Mixing `tx` and outer `prisma` **silently breaks** the transaction boundary.

### 2.4 Core entity relationships & uniqueness constraints

Agents must preserve these invariants when reading or writing data:

| Entity / relation | Invariant |
|-------------------|-----------|
| **User** | `id` is Clerk user id (string PK). `email` optional unique. |
| **Game → Field** | Every game has required `fieldId` → `Field`. Games may also carry `customLat` / `customLng` / `customLocation` for non-mapped locations. |
| **Game → Organizer** | `organizerId` → `User`. |
| **Game → Series** | Optional `seriesId` → `GameSeries` for recurring pickup games. |
| **Participation** | `@@unique([gameId, userId])` — one roster row per user per game. Status drives confirmed vs waitlist (`ParticipationStatus`). |
| **ChatRoom ↔ Game** | Group chat room id **equals** game id (see §2.3). |
| **ChatParticipant** | `@@unique([userId, chatId])` — duplicate join attempts throw **P2002** (see §4.2). |
| **Friendship** | `@@unique([userAId, userBId])` — canonical pair ordering required (see §3.3). |
| **FriendRequest** | `@@unique([requesterId, receiverId])`. |
| **UserRating** | `@@unique([gameId, raterId, targetId])` — one rating per rater→target per game. |
| **FavoriteField** | `@@unique([userId, fieldId])`. |

### 2.5 Fields, courts & spatial discovery

From frontend architecture (field/court UX):

- **Field catalog:** `Field` records power court/venue profiles (`next_app/src/app/fields/[id]/page.tsx`).
- **Map search:** `SearchMapComponent.tsx`, `FieldBusyChart.tsx` — Leaflet-based; **dynamic import with `ssr: false` only**.
- **Game booking:** Games bind to a `fieldId`; location display may use field coordinates or game-level `customLat`/`customLng` when not tied to a mapped field.
- **Spatial indexes:** `Field` uses `@@index([lat, lng])` and `@@index([available, lat, lng])` for availability/geo queries — preserve index-friendly filters when adding field search logic.

### 2.6 User write safety (database side of auth pitfall)

When upserting `User` from authenticated requests, **never blind-overwrite** `name` / `imageUrl` with Clerk fallback values (`name === userId`). Use undefined-ignore pattern documented in §3.2.

## 3. SECURITY, AUTHENTICATION & PITFALLS

### 3.1 Clerk authentication flow (Express)

- Middleware: `authenticateToken` in `server/utils/auth.js` wraps `@clerk/express` `requireAuth()`.
- On success: fetches Clerk user via `clerkClient.users.getUser(userId)` and sets:

  `req.user = { id, name, avatar, isAdmin: false }`

- **Optional auth:** `attachOptionalUser` — no `Authorization` header → continue as guest; invalid token → guest (not 401).
- **Socket auth:** `clerkClient` is exported for Socket.io middleware in `server/index.js`. Unauthenticated sockets receive `'Unauthorized: Please login'` / `'Unauthorized access to room'`.

### 3.2 The Clerk fallback overwrite bug (critical)

When Clerk API fails or profile is sparse post-signup, middleware falls back to:

```javascript
req.user = { id: userId, name: userId, avatar: null, isAdmin: false };
```

An unsafe upsert then **overwrites** the user's real display name with their Clerk id string and nulls their avatar:

```javascript
// UNSAFE — DO NOT REPLICATE
await prisma.user.upsert({
  where: { id: req.user.id },
  update: { name: req.user.name, imageUrl: req.user.avatar },
  create: { id: req.user.id, name: req.user.name, imageUrl: req.user.avatar }
});
```

**Required pattern** — treat fallback name as "do not update":

```javascript
const isNameFallback = req.user.name === req.user.id;

await prisma.user.upsert({
  where: { id: req.user.id },
  update: {
    name: isNameFallback ? undefined : req.user.name,
    imageUrl: (isNameFallback || !req.user.avatar) ? undefined : req.user.avatar
  },
  create: {
    id: req.user.id,
    name: isNameFallback ? null : req.user.name,
    imageUrl: isNameFallback ? null : req.user.avatar
  }
});
```

Prisma skips `undefined` fields on update — existing DB name/avatar are preserved.

### 3.3 Friendship ordering constraint

`Friendship` enforces `@@unique([userAId, userBId])`. Always sort ids before query/write:

```javascript
function orderPair(a, b) {
  return a < b ? [a, b] : [b, a];
}
```

Prevents reciprocal duplicates (A→B and B→A) and constraint crashes.

### 3.4 PII & privacy

- User model includes optional `email`, `phone`, `birthDate`, `city` and privacy flags (`privacyFriends`, `privacyGames`, `privacyMessages` with `PrivacyLevel`).
- **Never log** raw JWTs, Clerk secrets, or full profile payloads in production paths.
- **Env secrets:** `CLERK_SECRET_KEY`, `DATABASE_URL`, `DIRECT_DATABASE_URL`, `FIREBASE_PRIVATE_KEY` — never commit; normalize Firebase keys per §4.1.

### 3.5 Admin & legacy auth dead ends

- `req.user.isAdmin` is **hardcoded false** everywhere — admin-gated features are stubs, unreachable in production.
- Do not wire new features to legacy `server/routes/auth.js` or `dataManager.js`.

## 4. REAL-TIME & STATE MANAGEMENT

### 4.1 Firebase private key double-escaping

Coolify and similar platforms may double-escape PEM newlines (`\\n`). Always normalize:

```javascript
privateKey: process.env.FIREBASE_PRIVATE_KEY
  ? process.env.FIREBASE_PRIVATE_KEY.replace(/\\+n/g, '\n')
  : undefined
```

Required for Firebase Admin signature checks (push notifications via `server/services/notificationService.js`).

### 4.2 Chat participant idempotency (P2002)

Concurrent joins or double-taps on `chatParticipant.create` can hit `@@unique([userId, chatId])`:

```javascript
try {
  await prisma.chatParticipant.create({ data: { userId, chatId } });
} catch (e) {
  if (e.code !== 'P2002') throw e; // already in room → success
}
```

### 4.3 Socket.io event contract (backend → clients)

Key broadcasts agents must preserve when mutating games/chat:

| Event | Source / usage |
|-------|----------------|
| `game:updated` | Per-user payload via `io.to(\`user_${uid}\`).emit(...)` in `gameService.js` — roster/matchmaking state |
| `game:created` / `game:deleted` | List feeds + `GameUpdateContext` subscribers |
| `series:created` / `series:deleted` | Recurring game series lifecycle |
| `chat:sync` | Per-recipient chat list sync (`io.to(\`user_${recipientId}\`).emit('chat:sync', ...)`) — **not** general notification feed |
| `message`, `messageUpdated`, `messageDeleted`, `messageReaction` | Chat thread updates |
| `presence:update` | Online status |

**Notification rule:** Chat messages use `chat:sync` + message counters — **never** duplicate chat content into the general notification feed (`notificationService.js`).

**API mutation rule:** Any game mutation route must return the **full updated game object** in the HTTP response **and** emit the matching socket event — frontend merge logic depends on both channels.

### 4.4 Frontend live-state architecture

| Mechanism | Role |
|-----------|------|
| `SocketContext.tsx` | Single global Socket.io connection; gated on Clerk `useAuth()` loaded + signed in |
| `GameUpdateContext.tsx` | Pub/sub for `game:*` / `series:*` — subscribe via `useGameUpdateListener`, never raw `socket.on` for these |
| `ChatContext.tsx` | Chat list + `messagesCache` keyed by room id (shared across `Chat.tsx` and `FloatingChatWindow`) |
| `NotificationCountersContext.tsx` | Navbar badge counts |
| `useSyncedGames.ts` | List-level game feed; dedupes `game:created` by id before insert |

### 4.5 Join / leave / matchmaking merge contract

Every mutating game action (join, leave, approve, reject, waitlist-confirm):

1. Client `fetch` awaits **full updated game** in response body.
2. Success callback merges: `setGame(prev => ({ ...prev, ...normalizeIncomingGame(updated) }))` — **never** `setState(updated)` alone.
3. Same component listens for `game:updated` socket and applies **identical** merge — idempotent if HTTP and socket both arrive.
4. List state dedupes: `if (prev.some(g => g.id === normalizedGame.id)) return prev`.

New mutations **must** follow all four steps so `chatRoomId` and other fields are not clobbered to `undefined`.

### 4.6 Chat optimistic updates

In `useChatLogic.ts`: reconcile server echo by **replacing** on `id || tempId`, never blind append. Render keys: `key={m.id || m.tempId}`.

### 4.7 Socket listener cleanup & Clerk stability

**Every** `socket.on(...)` / context subscribe / `setTimeout` in `useEffect` **must** return cleanup (`socket.off`, unsubscribe, `clearTimeout`). Effect deps must include route-scoped ids (`gameId`, `roomId`).

**Clerk `getToken` instability:** Freeze via ref pattern (see `useNotifications.ts` / `realtime_rules.md`) — do not put `getToken` directly in effect dependency arrays (causes refresh loops).

Applies to **React web** (`next_app/`) and **React Native** clients per shared SocketManager discipline.

### 4.8 Troubleshooting live sync

If UI does not update after a mutation:

1. Confirm backend emitted `game:updated` or `chat:sync` with full payload.
2. Confirm client merge uses `normalizeIncomingGame()` / `mapGameTimezones()`.
3. Confirm `NEXT_PUBLIC_SOCKET_URL` / Clerk token resolved before `io(...)` connect.
4. Do **not** "fix" CORS by disabling `credentials` or using `no-cors` on fetch — fix `server/index.js` CORS config instead.

## 5. PRODUCTION GUARDRAILS & OUT OF SCOPE


Tier 1 agents must document these commands for Tier 2 execution when terminal access is unavailable.

### 5.2 In-scope vs out-of-scope boundaries

#### In scope (default MAMS tasks)

- Express API routes, services, Prisma schema/migrations, Socket.io handlers, Firebase push wiring.
- Next.js App Router pages/components/hooks/contexts under `next_app/src/**`.
- Jest tests under `server/` for roster, lottery, auth behavior.
- Field/game/chat/series/friendship flows described in this document.

#### Out of scope (do not modify unless Task Contract explicitly overrides)

| Area | Path / topic | Reason |
|------|--------------|--------|
| **Standalone mobile app** | `mobile_app/` (Expo Router, React Native, NativeWind) | Separate client layer; realtime_rules treats RN as distinct from Next.js PWA patterns |
| **Legacy JSON auth store** | `server/routes/auth.js`, `server/utils/dataManager.js` | Dead code — not Clerk/Prisma |
| **Admin features** | Anything gated on `req.user.isAdmin` | Permanently stubbed / unreachable |
| **External payment / billing flows** | Not defined in agent constitution | No documented payment integration — requires explicit product task |
| **Infrastructure secrets & deploy plumbing** | Coolify/Render/Vercel env provisioning, Neon project creation | Environment must exist; agents configure code only |
| **Bulk field CSV import scripts** | `server/scripts/importFieldsFromCsv.js`, seed utilities | One-off ops unless task targets them explicitly |
| **MAMS microservice itself** | `mams-core-service/` (sibling repo) | JoinUp workspace rules govern JoinUp code only |

### 5.3 Change discipline

- **Schema / shared CSS variables:** Treat like migrations — deliberate, full-app visual check.
- **PWA service worker:** After `next.config.ts` PWA changes, verify regenerated `public/sw.js`; clear `.next` and unregister SW when debugging stale assets.
- **CORS / credentials:** Server-side fix only; never weaken client fetch security flags.
- **New dependencies:** Tier 2 installs only; justify in narrative summary.

### 5.4 Escalation triggers

Escalate to human review when:

- Task requires mobile_app changes but contract only mentions "frontend".
- Clerk fallback overwrite pattern cannot be applied (new User write path).
- Migration needs production `directUrl` access not available in sandbox.
- Real-time contract change affects multiple clients (web + mobile) simultaneously.
